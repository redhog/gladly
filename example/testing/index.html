Gladly WebGL Crash Tests

Open each test one at a time. Report which ones crash the entire browser (not just the tab).

  • 01 — desynchronized: true context
    Creates a WebGL2 context with { desynchronized: true } and runs a simple render loop.
  • 02 — Long-running single draw call
    Fullscreen quad with a 500K-iteration fragment shader. Tests Windows TDR threshold (~2s GPU timeout).
  • 03 — Many sequential FBO passes
    64 sequential framebuffer ping-pong passes with texture reads. Simulates FFT compute pipeline.
  • 04 — 1M-point additive blend scatter
    Draws 1 million points with additive blending. Simulates histogram scatter pass.
  • 05 — RGBA+FLOAT texImage2D format
    Calls texImage2D with RGBA (not RGBA32F) as internal format + FLOAT type. Tests invalid WebGL2 format handling.
  • 06 — Gladly ComputePipeline data generation
    Runs the full 1 M-point GPU data generation (13 columns) and CPU readback in isolation — no plot rendering. Times each column readback individually to pinpoint which step causes crashes.
  • 07 — Gladly 5 concurrent plot initializations
    Creates 5 plots simultaneously (3 scatter + 2 histograms with 1 M additive-blend scatter each). If 06 passes but 07 crashes, the bug is concurrent GPU command pile-up across multiple plots/contexts.
  • 08 — PlotGroup + bivariate colorscale (mirrors tab1)
    PlotGroup with 2 linked scatter plots using 1 M points and a 2D bivariate colorscale. This is the ONLY plot that runs immediately at main-example page load — if this crashes, tab1 is the culprit.
  • 09 — TileLayer + 1 M scatter overlay (mirrors tab4)
    OSM XYZ tile background (EPSG:3857→4326 reprojection) plus 1 M-point scatter. Tests tile HTTP fetch + texture creation + tessellated quad rendering. If 07 passes but 09 crashes, TileLayer is the culprit.
  • 10 — LinesLayer + MultiLineLayer (mirrors tab2)
    Scatter + multi-line layer with 1 M data points and a filterbar axis. Tests instanced segment rendering — a different GPU workload from scatter. If 07 passes but 10 crashes, LinesLayer is the culprit.
  • 11 — 8 WebGL contexts created simultaneously, updated sequentially
    Creates 8 Plot objects (WebGL contexts) all at once, then calls plot.update() on each one at a time. Separates context-creation cost from rendering cost. Compare with test 07 (concurrent updates).
  • 12 — ONLY a 2D colorbar, no scatter (isolates ColorbarLayer2d shader)
    No scatter layers — only a ColorbarLayer2d float. ColorbarLayer2d always has colorAxes2d set, so all 9 registered 2D colorscale functions are reachable in its shader. If this crashes, the ColorbarLayer2d shader alone is the culprit.
  • 13 — Bivariate scatter (vData2) with 1D colorbars only, no 2D colorbar
    Scatter with both vData and vData2, colorbars: []. Because vData2 is present, colorAxis2dQuantityKinds is set on the layer (ScatterShared.js:30), making map_color_s_2d and all 9 2D colorscale functions reachable in the scatter shader — even though both cs_a and cs_b are 1D indices at runtime.
  • 14 — Only bilinear4corner registered (simplest 2D colorscale, no branches)
    Does NOT import src/index.js. Registers ONLY bilinear4corner (pure bilinear math, no if-branches). The ColorbarLayer2d shader has just 1 2D colorscale function. If this crashes, any 2D colorscale at all is enough to trigger the D3D11 HLSL failure.
  • 15 — Only moreland_5x5 registered (most complex 2D colorscale, 25 branches)
    Same as 14 but registers moreland_5x5 instead (25 nested if/else-if branches with int indexing). Compare with 14: if 14 passes but 15 crashes, it is GLSL complexity (not mere presence of 2D functions) that kills the D3D11 compiler.
